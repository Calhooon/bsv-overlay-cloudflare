# Cross-Validator Oracle (Layer 3)

Runs **both** our Rust `UHRPTopicManager::validate_uhrp_output` AND the
TypeScript `UHRPTopicManager.identifyAdmissibleOutputs` from
`@bsv/overlay 0.6.0` + `@bsv/sdk 1.10.1` (the exact pinning that
`nanostore.babbage.systems` and `overlay-us-1.bsvb.tech` deploy) over the
same corpus of locking scripts, and asserts both implementations agree
on every admit/reject decision.

## Why a Layer 3 oracle?

| Layer | What it proves | What it misses |
|-------|----------------|----------------|
| 1: byte-exact PushDrop goldens | Field encoding matches @bsv/sdk byte-for-byte | Doesn't prove the validator interprets those bytes the same way |
| 2: live JSON harness | Response shape matches nanostore | Hits live infra; tests the network path, not the rule |
| **3: cross-validator** | **Admission rule itself agrees on the same input bytes** | Requires Node on PATH |

Concretely: bsvb could change `UHRPTopicManager.ts` to forbid IP-literal
hosts in `field[2]`. The PushDrop bytes wouldn't change. Layer 1 would
still pass. Layer 2 would only catch it on URLs we happen to try. Layer 3
would surface the divergence on the very first run because the same
corpus produces different verdicts.

## Architecture

```
+-----------------------------------------------------------+
|  cross_validator.rs (Rust integration test)               |
|  ───────────────────────────────────────────────          |
|  1. Build corpus:                                         |
|       • 12 hand-crafted cases (PushDrops with various     |
|         valid/invalid shapes — see "Adding cases" below)  |
|       • N cases loaded from                               |
|         bsv-storage-cloudflare/tests/parity/fixtures/     |
|           parity/admission_transcript.json                |
|  2. For each case: run                                    |
|       UHRPTopicManager::validate_uhrp_output(output, now) |
|     and record admit/reject.                              |
|  3. Spawn `node cross_validate.mjs`, send entire corpus   |
|     as JSON on stdin, read verdicts from stdout.          |
|  4. Assert label-by-label that Rust admit == Node admit.  |
+-----------------------------------------------------------+
                            |
                            | spawn (one subprocess for the whole corpus)
                            v
+-----------------------------------------------------------+
|  cross_validate.mjs (in bsv-storage-cloudflare/tests/     |
|    parity/ts-fixtures/)                                   |
|  ───────────────────────────────────────────────          |
|  For each case:                                           |
|    • Wraps locking_script_hex in a synthetic Transaction  |
|      (1 placeholder input + 1 output under test).         |
|    • Calls tx.toBEEF(true) to produce the BEEF bytes      |
|      identifyAdmissibleOutputs expects.                   |
|    • Invokes new UHRPTopicManager().                      |
|        identifyAdmissibleOutputs(beef, [])                |
|      from tm_uhrp_local.mjs (verbatim copy of bsvb's      |
|      UHRPTopicManager.ts pinned to @bsv/* node_modules).  |
|    • Captures console.error() output for reject reasons.  |
|  Writes {verdicts: [...]} JSON to stdout.                 |
+-----------------------------------------------------------+
```

The shim is invoked **once per test run** with the entire corpus to
avoid Node startup overhead (~100ms per case adds up). The shim is
stateless — safe to run from `cargo test` parallelism.

## Skip conditions

The test passes (with `SKIPPED: ...` on stderr) when:

1. `node` is not on `$PATH`.
2. `node_modules` is missing in `bsv-storage-cloudflare/tests/parity/ts-fixtures/`.

CI containers without Node still pass the rest of the suite.

## Files

| File | Purpose |
|------|---------|
| `cross_validator.rs` (this dir) | Rust driver — builds corpus, runs Rust validator, spawns Node shim, asserts agreement |
| `../../../../bsv-storage-cloudflare/tests/parity/ts-fixtures/cross_validate.mjs` | Node shim — wraps cases in BEEF, runs TS UHRPTopicManager, emits verdicts JSON |
| `../../../../bsv-storage-cloudflare/tests/parity/ts-fixtures/tm_uhrp_local.mjs` | Verbatim copy of bsvb's UHRPTopicManager.ts (translated to JS, no semantic changes) |
| `../../../../bsv-storage-cloudflare/tests/parity/fixtures/parity/admission_transcript.json` | Corpus inputs generated by `generate.mjs` |

## Running

```bash
# From rust-overlay/ root:
cargo test -p overlay-discovery --test cross_validator

# Verbose output (shows case count + reject reasons):
cargo test -p overlay-discovery --test cross_validator -- --nocapture

# First-time setup (if SKIPPED on node_modules):
cd ../bsv-storage-cloudflare/tests/parity/ts-fixtures && npm install
```

Expected: `cross_validator: 14 cases, 4 admitted, 10 rejected, 0 divergences`.

## Adding cases

### Hand-crafted (in this file)

Add a new entry to `build_hand_crafted_corpus()` in `cross_validator.rs`.
Pattern:

```rust
cases.push(CorpusCase {
    label: "hand_<short_name>".into(),
    locking_script_hex: script_hex(&make_signed_uhrp_output(
        &admin, hash, url, expiry, content_length,
    )),
    satoshis: 1,
    expected_admit: <true|false>,
    note: "<what this exercises>".into(),
});
```

For invalid-shape cases, use `make_raw_uhrp_output_with_junk_sig(fields)`
which builds a 6-pushed-field PushDrop with a junk signature so both
validators walk past the structural-shape checks before hitting the
specific gate you want to test.

The `expected_admit` field is the corpus author's prediction. **Cross-
validator agreement is the oracle**, not `expected_admit` — if Rust and
TS agree but disagree with `expected_admit`, the test logs a
`CORPUS_NOTE` to stderr but does NOT fail. Update the note (and possibly
the field) in that case.

### Generated (regenerate transcript)

```bash
cd ~/bsv/bsv-storage-cloudflare/tests/parity/ts-fixtures
node generate.mjs
# Produces ~/bsv/bsv-storage-cloudflare/tests/parity/fixtures/parity/admission_transcript.json
```

This file is consumed by `load_transcript_corpus()` in `cross_validator.rs`.
Owned by another agent — coordinate before adding fixtures.

### Real on-chain BEEF

Currently NOT included in the corpus — the cross-validator's input format
is `locking_script_hex`, not BEEF. To add a real on-chain advert, decode
the BEEF, extract the relevant output's locking script hex, and add it as
a hand-crafted case with a `real_onchain_<txid_short>` label.

```bash
# Inspect a captured BEEF
xxd /tmp/our_uhrp_0.beef | head
# Decode tx, pick the UHRP output, hex-encode its locking script,
# add to build_hand_crafted_corpus().
```

## Divergence triage

When the test fails:

```
DIVERGENCE 'hand_<label>': rust=<true|false> node=<true|false> (expected=<true|false>) — note: <...> — node_reason: "<...>"
```

This is a real admission-rule drift bug. Triage:

1. **Read `node_reason`** — it's the captured `console.error('Error with output', i, error)` from the TS validator.
2. **Re-run Rust validator manually** with `validate_uhrp_output(output, now)` and inspect the `Err(reason)` to see where Rust rejects.
3. **Compare to UHRPTopicManager.ts** at `~/bsv/uhrp-services/backend/src/topic-managers/UHRPTopicManager.ts`. The TS source is the canonical reference.
4. **Fix the side that's wrong**:
   - If the TS rule changed (e.g. tightened URL parsing): update Rust to match.
   - If our Rust impl drifted: revert.
5. Add a regression note in the case's `note` field.

Never just delete a divergent case — it's a real bug somewhere.

## Pinning notes

`tm_uhrp_local.mjs` is a **hand copy** of bsvb's `UHRPTopicManager.ts` +
`isTokenSignatureCorrectlyLinked.ts`. We don't import the upstream
package because (a) `uhrp-services` isn't on npm and (b) pinning lets us
prove drift independently of upstream's release cadence.

Re-sync procedure when bsvb updates:

1. Diff `tm_uhrp_local.mjs` vs `~/bsv/uhrp-services/backend/src/topic-managers/UHRPTopicManager.ts`.
2. Apply the upstream changes line-for-line. NO refactoring.
3. Bump `_meta.note` in any regenerated transcript so reviewers notice.
4. Run `cargo test -p overlay-discovery --test cross_validator`. If new
   divergences appear, see "Divergence triage".

The `@bsv/sdk 1.10.1` and `@bsv/overlay 0.6.0` versions in
`ts-fixtures/package.json` must continue to match what nanostore /
overlay-us-1 deploy. Bump only when they bump.
